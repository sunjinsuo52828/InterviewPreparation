2. Cache Consistency (DB vs Redis) High Frequency

How to keep Redis and MySQL in sync?

Don't update the cache when DB changes. If two threads update concurrently, you might get dirty data (Race Condition).

  1. Read: Get from Cache -> Miss? -> Get from DB -> Set Cache.
  2. Write: Update DB -> Delete Cache.

If you delete cache before DB commit, a reader might load old data back into cache. Solution: Delete once, wait 500ms, delete again.

3. Distributed Locks Medium Frequency

Using Redis to ensure only one service instance executes a task.

// Atomic Command (Set if Not Exists + Expiry)
SET lock_key unique_id NX PX 30000
  • NX: Only set if key doesn't exist (Mutex).
  • PX 30000: Auto-expire after 30s (Prevent Deadlock if server crashes).
  • Value (unique_id): Ensure you only delete your own lock (Validation).

Redlock Algorithm: For distributed Redis (Cluster), acquire locks on N/2 + 1 nodes.

4. Big Key & Hot Key Problems Medium Frequency

üêò Big Key (Value is too large)

  • Example: A Hash with 1 million fields.
  • Fix: Split into multiple keys (hash_1, hash_2...).

üî• Hot Key (Too much traffic on one key)

5. Redis 6.0 Threading Model New Topic

Question: "Is Redis single-threaded?"

Answer:

  • Command Execution: Still Single-Threaded (No locks needed, atomic).
  • Network I/O: Now Multi-Threaded in Redis 6.0.

Why? The bottleneck was network packet parsing, not CPU execution.
