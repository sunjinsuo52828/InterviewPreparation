Spring Bean Lifecycle (Singleton)

Occurs during refresh() -> finishBeanFactoryInitialization()

1. Instantiation Constructor
Spring creates the bean instance using the constructor (or factory method).
2. Populate Properties Setter / Field Injection
Spring injects dependencies (@Autowired, @Value) into the bean.
3. Aware Interfaces BeanNameAware, BeanFactoryAware
If the bean implements Aware interfaces, Spring injects infrastructure objects (ID, Factory, Context).
4. BeanPostProcessor (Before) postProcessBeforeInitialization
Critical Hook: Modify the bean instance before initialization (e.g., @PostConstruct handling).
5. Initialization @PostConstruct, InitializingBean
Custom init logic runs here. (1. @PostConstruct, 2. afterPropertiesSet, 3. init-method).
6. BeanPostProcessor (After) postProcessAfterInitialization
Critical Hook: Create AOP Proxies here. If the bean needs transaction/async, it is wrapped in a Proxy now.
7. Ready to Use
Bean is fully initialized and stored in the Singleton Cache.
8. Destruction @PreDestroy, DisposableBean
When context closes.

Q: How does @EnableAutoConfiguration work?
It uses @Import(AutoConfigurationImportSelector.class) to load configurations listed in META-INF/spring.factories (or META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports in newer versions).
Q: ApplicationRunner vs CommandLineRunner?
Both run after startup.
- CommandLineRunner receives raw String[] args.
- ApplicationRunner receives parsed ApplicationArguments object (easier to access flags like --foo=bar).
Q: How to create a Custom Starter?
1. Create a AutoConfiguration class.
2. Add conditional annotations (@ConditionalOnClass).
3. Register it in spring.factories.
