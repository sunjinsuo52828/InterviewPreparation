Docker & OpenShift Architecture

Containerization vs Orchestration: The Big Picture

What are they?

Docker is a platform for developing, shipping, and running applications in containers. It solves the "it works on my machine" problem.

OpenShift is an enterprise Kubernetes platform (PaaS) built by Red Hat. It adds developer-friendly features (Source-to-Image, Routes, Web Console) on top of raw Kubernetes.

The Dockerfile

The blueprint for building a Docker image.

# 1. Base Image FROM node:14-alpine # 2. Working Directory WORKDIR /app # 3. Copy Dependencies COPY package*.json ./ # 4. Install Dependencies RUN npm install --production # 5. Copy Source Code COPY . . # 6. Expose Port EXPOSE 3000 # 7. Start Command CMD ["node", "server.js"]

Key Commands

  • docker build -t myapp:v1 . - Build an image from Dockerfile.
  • docker run -d -p 8080:3000 myapp:v1 - Run container in background, map port 8080 to 3000.
  • docker ps - List running containers.
  • docker exec -it <id> sh - Enter a running container.
  • docker logs <id> - View container logs.

Step-by-Step:

  1. Route: Accepts traffic for a specific hostname (e.g., www.example.com) and forwards it to the associated Service.
  2. Service: Acts as an internal load balancer. It tracks the IP addresses of all healthy Pods matching its selector.
  3. Pod: The actual container processing the request.

Deep Dive: Under the Hood

1. How Router maps Host to Service

The OpenShift Router is typically a container running HAProxy.

  • Watch: The Router watches the API Server for Route objects.
  • Config Update: When you create a Route (api.example.com), the Router detects this event.
  • Map File: It updates its internal HAProxy map file:
    api.example.com be_frontend_svc
  • Direct to Pods: For performance, the Router usually bypasses the Service ClusterIP and load balances directly to the Pod IPs listed in the Service's Endpoints.

Q: What if a Pod dies and restarts? Does it notify the Service?

No, the Pod does not notify anyone. Instead, the Kubernetes Control Plane (Endpoints Controller) is constantly watching.

  1. Pod Dies: The Controller detects the Pod is gone or not "Ready". It immediately removes the Pod's IP from the Service's Endpoints list. Traffic stops going there.
  2. Pod Restarts: A new Pod is created with a new IP.
  3. Readiness Probe: K8s waits for the readinessProbe to pass (e.g., `GET /health` returns 200).
  4. Update: Once Ready, the Controller adds the new IP to the Endpoints list. The Service (via iptables) now routes traffic to the new Pod.

This is "Service Discovery" in action: The Service abstraction ensures clients don't care about Pod IPs changing.

The `oc` CLI

Similar to `kubectl`, but with extra commands for OpenShift features.

  • oc login - Authenticate to the cluster.
  • oc new-project my-project - Create a new namespace/project.
  • oc new-app nodejs~https://github.com/user/repo - Magic command: Creates BuildConfig, DeploymentConfig, Service, and ImageStream from source.
  • oc get pods - List pods.
  • oc rsh <pod> - Remote shell into a pod (like docker exec).
  • oc expose svc/frontend-svc - Create a Route for a service.

Interview Questions

Docker Q1: Docker Image vs Container?

An Image is a read-only template with instructions for creating a container (like a Class). A Container is a runnable instance of an image (like an Object). Images are built from layers.

OpenShift Q2: What is a Pod?

The smallest deployable unit in Kubernetes/OpenShift. A Pod represents a single instance of a running process. It can contain one or more containers that share storage (volumes) and network (IP address).

OpenShift Q3: Route vs Ingress?

Route is an OpenShift-native concept created before Kubernetes Ingress existed. It exposes a Service to the outside world via a hostname. Ingress is the standard Kubernetes equivalent. OpenShift supports both, but Routes are easier to use in OCP.

OpenShift Q4: What is Source-to-Image (S2I)?

A toolkit in OpenShift that allows you to build reproducible Docker images from source code. You don't need to write a Dockerfile; S2I combines your source code with a "Builder Image" (e.g., Node.js builder) to create the final app image.

Docker Q5: Explain Docker Layers & Caching.

Each instruction in a Dockerfile (RUN, COPY) creates a layer. Layers are cached. If you change a line, that layer and all subsequent layers must be rebuilt. Optimization tip: Copy `package.json` and run `npm install` before copying the rest of the source code to cache dependencies.
